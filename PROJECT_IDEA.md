# PROJECT_IDEA.md — Bot_HH_Habr

## 1. Проблема

Рекрутер или владелец бизнеса с активными вакансиями на Хабр Карьере и HH.ru получает
**до 250 откликов в день**. Ручной просмотр каждого — 2–4 минуты, итого 8–16 часов в день.
Из 250 откликов реально подходящих по ключевым критериям (гражданство РФ + опыт ≥5 лет) —
10–30%. Остальные 70–90% — нерелевантный поток, который тратит время и внимание.

Email-уведомления от платформ отключены — их слишком много, они превращаются в спам.

## 2. Решение

Автономный бот-фильтр, работающий без участия пользователя:

```
Хабр Карьера / HH.ru
    ↓ (каждые 5 минут, polling)
Парсинг новых откликов
    ↓
Извлечение данных резюме (гражданство, опыт)
    ↓
Фильтрация по критериям
    ↓ ✅ подходит / 🟡 данных нет / ❌ не подходит
Уведомление в Telegram-канал (только ✅ и 🟡)
    ↓
Запись в Google Sheets (живая таблица HR)
    ↓
Архив всех откликов в Supabase (история + пересмотр фильтров)
```

Результат: рекрутер видит только релевантных кандидатов. Время просмотра — секунды на карточку.

## 3. Почему сейчас

- AI-driven parsing резюме стал доступен без специальных ML-моделей
- Supabase + Node.js позволяют поднять инфраструктуру за 1 день
- Хабр и HH не предоставляют встроенной фильтрации по гражданству + опыту одновременно
- Конкурентная среда: массового притока низкоквалифицированных откликов стало больше

## 4. Целевая аудитория

**Основная:** малый бизнес и стартапы с 1–5 активными вакансиями, ищущие опытных
разработчиков с гражданством РФ. Нет HR-отдела, найм ведёт CTO или основатель.
Текущий инструмент: ручной просмотр кабинета Хабр/HH раз в день.

**Вторичная:** HR-специалисты средних компаний, перегруженные потоком откликов.
Ищут автоматизацию без дорогих ATS-систем.

## 5. Архитектура

```
┌─────────────────────────────────────────────┐
│                 ИСТОЧНИКИ                    │
│  Хабр Карьера (cookie-парсинг)   HH.ru API  │
└───────────────────┬─────────────────────────┘
                    │ polling каждые 5 мин
┌───────────────────▼─────────────────────────┐
│            ВОРКЕР (Node.js 20, ESM)          │
│  ┌──────────┐ ┌──────────┐ ┌─────────────┐  │
│  │ habr.js  │ │  hh.js   │ │  filter.js  │  │
│  └──────────┘ └──────────┘ └──────┬──────┘  │
│                                   │         │
│  ┌────────────────────────────────▼──────┐  │
│  │           poller.js (оркестратор)     │  │
│  └────────────────────────────────────┬──┘  │
└───────────────────────────────────────┼─────┘
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
  ┌──────────┐ ┌─────────┐ ┌────────┐
  │ Telegram │ │ Sheets  │ │Supabase│
  │  канал   │ │ Google  │ │  БД    │
  └──────────┘ └─────────┘ └────────┘
```

**Стек:**
| Компонент | Технология | Обоснование |
|---|---|---|
| Runtime | Node.js 20+ ESM | Async/await, нативный fetch, долгоживущий процесс |
| Парсинг HTML | cheerio | Быстрый jQuery-like парсер, без headless-браузера |
| HTTP | axios | Retry, timeout, interceptors для cookie-сессий |
| БД | Supabase (PostgreSQL) | Self-hosted, RLS, realtime, готовый JS SDK |
| Уведомления | Telegram Bot API | Inline-кнопки, Markdown, мгновенная доставка |
| Таблица | Google Sheets API v4 | Живая таблица, доступна HR с телефона |
| Процесс-менеджер | PM2 / systemd | Автоперезапуск, логи, memory limit |
| Деплой | VPS vm7377 | Уже используется, рядом с другими сервисами |

## 6. Монетизация

Проект — внутренний инструмент, не SaaS. Прямой монетизации нет.
Косвенная ценность: экономия 4–8 часов времени ключевого человека в день.

## 7. Конкуренты

| Инструмент | Что делает | Чего не хватает |
|---|---|---|
| HH.ru встроенный фильтр | Фильтр по зарплате, должности | Нет фильтра по гражданству + опыту одновременно |
| Хабр Карьера фильтры | Базовые фильтры вакансий | Нет автоматических уведомлений с фильтрацией |
| Хантфлоу / Potok | Полноценные ATS | Дорого (от 15k/мес), избыточно для 1–5 вакансий |
| Zapier / Make | Автоматизация потоков | Нет встроенного парсинга Хабра, дорого при объёме |
| Ручной просмотр | — | 8–16 часов в день |

## 8. Фазы запуска

### MVP — Фаза 1 (текущая)
- Источник: Хабр Карьера (cookie-парсинг)
- Фильтр: гражданство РФ + опыт ≥5 лет
- Вывод: Telegram-канал + Google Sheets
- Хранение: Supabase (все отклики с флагами)
- Деплой: VPS vm7377, PM2

**Метрики успеха:** 0 пропущенных ✅-откликов, <5% ложных срабатываний 🟡

### Фаза 2
- Источник: HeadHunter API (OAuth 2.0)
- Тот же фильтр и пайплайн
- Единая таблица для обоих источников

### Фаза 3 (опционально)
- Веб-дашборд статистики (retention, конверсия фильтра)
- Telegram-команды: `/stats`, `/reprocess`, `/update_filter`
- Настройка фильтров без редактирования `.env`

## 9. Риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Cookie Хабра протухнет | Высокая (раз в 2–4 нед) | Алерт в TG при 401/redirect, быстрое обновление |
| Хабр изменит HTML-вёрстку | Средняя | Graceful error, alert, селекторы в одном месте |
| HH заблокирует по rate limit | Низкая | Задержки между запросами, один аккаунт |
| Гражданство не указано в резюме | Высокая (~40% откликов) | Категория 🟡 — не теряем, но помечаем |
| Google Sheets API квота | Низкая | Batch append, не чаще 1 запроса в 2 сек |
| Дублирование откликов при рестарте | Средняя | UNIQUE(source, external_id) в Supabase |

### Ключевые таблицы БД
- `applications` — все отклики (qualified: true/false/null)
- `oauth_tokens` — HH refresh/access tokens
- `processing_log` — история циклов, статистика

### Критерии фильтрации (текущие)
```javascript
citizenship === 'RU'          // ✅ пропустить
citizenship === 'OTHER'       // ❌ отклонить
citizenship === null          // 🟡 пометить
experience_years >= 5         // ✅ пропустить
experience_years < 5          // ❌ отклонить
experience_years === null     // 🟡 пометить
// Если оба null → 🟡 (отправить с двойной пометкой)
// Если один ❌ → ❌ (не отправлять)
```
